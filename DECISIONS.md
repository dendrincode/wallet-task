# Design Decisions, Trade-offs, and Next Steps

## Design Decisions

### Separate `users` table with a proper FK relationship
Rather than treating `userId` as a plain string stored directly on the wallet, I introduced a `users` table and made `wallets.user_id` a real foreign key. This enforces referential integrity at the database level and reflects the domain reality that a user exists independently of their wallet. The wallet is created on demand the first time a deposit or trade is made, so the user experience is unchanged — no explicit registration step is required.

### Dedicated `wallet_id` primary key
The `wallets` table has its own UUID primary key (`wallet_id`) generated by Hibernate, separate from the `user_id` foreign key. Using `userId` as the PK would have conflated the business identifier with the database identifier. This separation allows the schema to evolve — for example, supporting multiple wallets per user — without changing the public API.

### Pessimistic locking on trades only
Trade operations acquire a `PESSIMISTIC_WRITE` lock on the wallet row for the duration of the transaction. This makes the balance check and balance update atomic, which prevents concurrent overdrafts without any application-level coordination. Deposits do not use this lock because adding funds cannot cause an overdraft, so the cost of serialization is avoided there.

### Idempotency at the service layer with required keys
Every deposit and trade requires a UUID v4 idempotency key. Before applying any change, the service checks whether a transaction with that key already exists. If it does, the previous result is returned immediately without touching the wallet. This makes retry-safe clients simple: send the same request with the same key and you always get the same outcome. The key is validated at the controller boundary using Bean Validation rather than in the service layer, which keeps service logic focused on business rules.

### `BigDecimal` throughout
All balance and amount values are `BigDecimal` from the HTTP request body to the database column. Using floating-point types for money introduces rounding errors that compound over time. `BigDecimal` with scale 2 gives exact decimal arithmetic at the cost of slightly more verbose code.

### Wallet auto-creation
Wallets are created automatically on the first deposit or trade rather than requiring an explicit creation step. This reduces integration friction for clients and matches the exercise requirements. The `GET /wallets/{userId}` endpoint returns a zeroed-out response for unknown users without persisting anything, so reads are always safe.

### Spring Security wired but disabled
The security configuration is present and tested but disabled by default via a property flag (`wallet.security.enabled`). This means authentication can be turned on for any deployment without code changes, while keeping the local and test experience frictionless.

## Trade-offs

### Pessimistic vs. optimistic locking
Pessimistic locking serializes all trades on a given wallet, which eliminates overdrafts but reduces throughput under high concurrency on a single wallet. Optimistic locking would allow concurrent reads and retry on conflict, which is better for read-heavy workloads, but requires careful retry logic and can fail under sustained write contention. For a wallet service where correctness is more important than throughput on a single account, pessimistic locking is the safer default.

### `create-drop` DDL vs. migrations
The application uses `spring.jpa.hibernate.ddl-auto=create-drop` for local development and tests. This is convenient but not suitable for production: it destroys data on restart and gives no control over incremental changes. A migration tool would own the schema in any real deployment.

### H2 for integration tests vs. real PostgreSQL
Integration tests run against H2, which is fast and requires no external dependencies. The downside is that H2 does not behave identically to PostgreSQL — dialect differences, locking semantics, and certain DDL features differ. A test failure that only reproduces on PostgreSQL would not be caught in CI.

### One wallet per user
The `UNIQUE` constraint on `wallets.user_id` enforces one wallet per user at the database level. This matches the exercise scope but would need to be relaxed for a multi-wallet product (savings, trading, rewards, etc.).

### Idempotency key stored on the transaction row
The idempotency key is stored as a unique column on the `transactions` table. This is simple and correct but means the key is permanently tied to one transaction and cannot be reused across different operations. A more sophisticated design would store idempotency records separately with a TTL, allowing key expiry and reuse after a safe window.

## What I Would Do With More Time

### Schema migrations with Flyway
Replace `ddl-auto` with Flyway. Every schema change would be a versioned SQL script checked into the repository, giving a complete audit trail and safe rollout to production databases.

### Integration tests against real PostgreSQL with Testcontainers
Testcontainers can spin up a real PostgreSQL instance for each test run, eliminating the H2/PostgreSQL dialect gap. This catches issues like PostgreSQL-specific locking behavior, type coercions, and constraint differences that H2 silently ignores.

### Authentication and authorisation
Spring Security is already wired. I would add JWT-based authentication so that a user can only access their own wallet — currently any caller can read or modify any wallet by providing a `userId`.

### Event publishing for significant operations
Publish domain events (e.g., `DepositCompleted`, `TradeFailed`, `WalletFrozen`) to a message broker so downstream systems can react without polling. This enables webhook delivery, fraud detection triggers, and real-time dashboards without coupling them to the wallet service.

### Rate limiting
Protect the deposit and trade endpoints from abuse with per-user rate limits. This is especially important for idempotency: a client that retries too aggressively can still create lock contention even if it never produces duplicate transactions.

### Reporting and analytics endpoints
Add endpoints for lifetime summaries (total deposited, total traded, net balance over a period), transaction search with filters, and wallet activity windows. The data is already captured — it just needs to be surfaced.

### Caching for wallet reads
`GET /wallets/{userId}` is likely called far more than it is written. An in-process cache (Caffeine) or a distributed cache (Redis) with short TTL would reduce database load without introducing significant staleness risk.

### Multi-currency support with exchange-rate integration
`CurrencyType` is already an enum on the wallet. With more time I would add an exchange-rate service, validate that trade amounts are in the wallet's currency, and support cross-currency reporting in a base currency.
